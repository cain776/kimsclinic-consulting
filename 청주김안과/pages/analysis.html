[문제 1] Windows 전용 Fat Client 구조 Critical
업무 이동성 전무 · 배포 비용 누적 · 장애 파급력 큼

플랫폼 한계

  • C# WinForms 기반 — 설치된 Windows PC에서만 동작. Mac · 태블릿 · 모바일 불가
  • 재택/원격 근무 불가, 원내 단말 외 업무 접근 경로 전무
  • 비즈니스 로직이 클라이언트에 분산되어 버전 불일치 발생

배포 및 운영

  • 기능 추가·버그 수정 시 모든 단말에 수동 재설치
  • CI/CD 부재 — 빌드/배포가 전부 수동
  • 모놀리식 EXE 구조 — 일부 장애가 전체 프로그램 중단으로 이어짐
  • .NET Framework 버전 종속 — OS/런타임 업그레이드 호환성 리스크

[문제 2] 시스템 분산 · 데이터 이중 입력 Critical
동일 환자 정보를 3개 시스템에 반복 입력 → 오타 · 누락 · 통합 분석 불가

현재 구조

  • 성민CRM · MD소프트 · 종이차트 세 시스템이 물리적으로 분리
  • 환자 등록 시 동일 정보를 여러 시스템에 반복 입력
  • SOFTCRM / MCRM / IPLUS DB가 Linked Server로 하드 결합되어 분리 어려움

데이터 무결성 문제 (성민CRM 기반 병원에서 공통적으로 관찰되는 패턴 · 저희 타 병원 사례 기준)

  • CUSTOM_MEMO RTF 저장 — 검색 불가, 구조화 불가, 연동 불가
  • DECRYPTBYPASSPHRASE 암호화 필드 — 패스프레이즈 공유, 복호화 비용으로 검색 저하
  • 코드 체계 불일치 (예: 보험사 INS00X vs 0X)
  • EMPLOYEE.EMP_NUM 6자리 고정 — 스키마 변경 없이 확장 불가
  • MsgSndLst SMS 이력 단일 테이블 수백만건 규모 — 파티셔닝 없음, 더미 레코드 혼입
  • 사용이 중단된 레거시 테이블이 스키마에 남아 혼선 유발

* 실제 청주김안과 DB 상태는 파일럿 단계 "현황 분석"에서 정확히 확인 예정

근본 원인

  • 인덱스 전략 부재 — 현장에서 필요할 때마다 수동 추가
  • 수평 확장 불가 — 단일 MSSQL 인스턴스에 모든 부하 집중
  • 성능 측정 문화 부재 — p95/p99 지표 없이 감각적 튜닝
  • 캐싱 전략 부재 — 동일 조회 반복 시 매번 DB 히트
  • Slow Query 모니터링·알림 없음
[문제 4] 보안 · 컴플라이언스 취약 Critical

인증 · 접근 제어

  • DB 접속 정보 하드코딩 (소스/설정 파일 평문 저장)
  • RBAC 세분화 부족 — 직원 전원이 사실상 동일 권한
  • 감사 로그(Audit) 부재 — 누가 언제 어떤 환자를 조회·수정했는지 추적 불가
  • 단일 세션 정책 없음 — 한 계정 다중 로그인 허용
  • 세션 타임아웃 부재 — 로그인 상태 무제한 유지
  • 비밀번호 해시 알고리즘·복잡도·만료 정책 부재

데이터 보호

  • 주민번호 DECRYPTBYPASSPHRASE 패스프레이즈 공유
  • 내부망 통신 HTTPS/TLS 강제 부재
  • 개인정보 대량 조회 임계 알림 없음

장비 · 외부 연동

  • 검사장비 수기 입력 — 시력·굴절·안압·OCT 등 장비 값을 사람이 차트에 옮겨 적음
  • CTI 콜센터 미연동 — 전화 수신 시 고객 자동 조회 불가
  • SMS/알림 채널 제한적 — 카카오톡·이메일·푸시 미지원 또는 별도 연동 필요
  • 외부 API 연동 어려움 — 신규 마케팅 채널·보험사 API 연동마다 커스텀 개발
  • Webhook/이벤트 발행 부재 — 타 시스템 실시간 연동 불가

확장 비용

  • 신규 장비 · 서비스 추가 = 유상 커스텀 개발
  • 리포트 포맷 변경이 개발 작업 (사용자 커스터마이징 불가)
  • 병원별 맞춤 UI 변경 시 전체 재개발
[문제 6] 관측성 · 운영 체계 부재 High
장애 원인 파악 지연 · 사용자 제보로만 인지

백엔드 · 운영

  • 중앙 집중 로그·메트릭·트레이스 없음 — 에러는 사용자 제보로 인지
  • 단일 창구 부재 — 시스템마다 별도 업체·별도 계약·별도 SLA
  • 장애 원인이 어느 시스템인지 판단 자체에 시간 소요
  • 장애 이력 DB 없음 — 동일 이슈 반복 발생
  • 성능 베이스라인 없음 — "예전보다 느려졌다"를 정량 측정 불가

테스트 · 배포

  • 회귀 테스트 자동화 부재 → 배포 리스크 높음
  • 롤백 전략 없음 → 배포 실패 시 수동 복구
  • 업무 매뉴얼 부재 또는 구버전 → 신규 직원 온보딩 곤란
[문제 7] AI/AX 관점의 구조적 한계 High
현재 레거시 구조에서는 AX(AI Transformation) 구현 사실상 불가

데이터 파이프라인

  • 데이터 정제 파이프라인 없음 — AI 학습용 데이터셋 구축 불가
  • Vector Search 불가 — MSSQL에 pgvector 상당 기능 없음 → 유사도·임베딩 저장 어려움
  • JSONB·반구조 데이터 저장 불가 — LLM 출력(요약/태그/메타) 저장 공간 부족

애플리케이션 계층

  • LLM 연동 API 레이어 없음 — Fat Client에서 Claude/OpenAI 직접 호출 시 키 유출 리스크
  • 실시간 스트리밍 UI 불가 — WinForms에서 토큰 스트리밍 응답 구현 난이도 높음
  • 이벤트 기반 아키텍처 부재 — AI Agent의 Tool Call 연계 어려움
  • 로그 수집 체계 없음 — AI 이상 감지·드리프트 탐지 불가
